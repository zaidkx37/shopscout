Architecture Approaches

Pick the architecture that fits your needs. Recommended: Option A.

A

Simple Monolith (Recommended)

FastAPI + SQLite + APScheduler

Single Python process handles everything: API, scheduler, and review submission. APScheduler stores jobs in SQLite so they survive restarts.

React Dashboard → FastAPI → SQLite
                          ↓
                APScheduler → TrustooSubmitter → Shopify

Pros

  • Single process, single container, single deploy
  • SQLite = zero config, backup = copy one file
  • APScheduler persists jobs to DB, survives restarts
  • Perfect for single-user, single-store
  • ~500 lines of backend code total

Cons

  • One crash takes down API + scheduler together
  • Would need refactoring for 100+ concurrent users (SaaS)
B

Separated Services

FastAPI + Celery + Redis + PostgreSQL

API and worker are separate processes. Celery handles job queue with retry logic. PostgreSQL for structured data. Redis as message broker.

React Dashboard → FastAPI → PostgreSQL
                          ↓
                  Redis → Celery Worker → TrustooSubmitter
                          ↓
                Celery Beat (cron scheduler)

Pros

  • Battle-tested job queue with retries and monitoring
  • API and worker scale independently
  • SaaS-ready from day one

Cons

  • 4 processes to manage (API, worker, beat, Redis)
  • PostgreSQL + Redis = more infra, more config
  • Massive overkill for single-user internal tool
  • 2-3x more code and setup time
C

Minimal Script + Cron

FastAPI dashboard + OS-level cron + JSON files

Dashboard is just a UI for viewing/editing JSON configs. Actual submission is a standalone Python script triggered by system cron every N minutes.

React Dashboard → FastAPI → JSON files

OS Cron (every 15min) → submit_reviews.py → Trustoo API

Pros

  • Dead simple, almost no moving parts
  • Dashboard and submission are fully decoupled
  • Easy to debug (just run the script manually)

Cons

  • Cron is coarse (minimum ~1 min intervals)
  • No built-in retry or job tracking
  • State management via JSON files gets messy
  • Hardest to extend to SaaS later
